Frequently asked questions


General

GenomeCAT reports a problem with the connection to MySQL. What could be the causes and how to solve the problem?

One reason for a failed connection to MySQL could be a problem with the configuration of the MySQL database.
Problems with MySQL configuration have never been observed by us when MySQL has been installed together with GenomeCAT, but can occur when users deselect MySQL from the GenomeCAT installation package, because of a pre-installed version of MySQL. However, in this case GenomeCAT only works if MySQL is always running (as a background process) and binaries are available.
Problems with MySQL can also arise when GenomeCAT is installed on a computer with an existing GenomeCAT installation. Uninstall the old version before.
Attention: "Uninstall" GenomeCAT also deletes the GenomeCAT database and all of its content! If you just want to update to the latest software version, it is better to take advantage of the automatic update feature provided by GenomeCAT.
When MySQL worked fine after installation, but loses connection at some later time point a connection time out may have occurred. This could happen when GenomeCAT is running in idle state for a longer time. In this case restarting the program should solve the problem.

Why does GenomeCAT shows the start screen, but does not complete starting?

GenomeCAT depends on an active MySQL data base running in the background. If you use a MySQL environment that has not been installed in the course of the GenomeCAT installation procedure please ensure that MySQL is running. As of GenomeCAT version 4.3 this problem should no longer exist for Windows7 and Linux systems usually start their MySQL server automatically.

Why does the graphical user interface does not show all windows or doesn't start at all?

Data on the configuration of how windows are displayed in GenomeCAT are stored in the local user directory. If you encounter the problems described above, deletion of the following directory can solve the problem.

You can find this directory on Windows7 computers in: C:\Users\XXX\AppData\Roaming\.genomecatpro\dev
On Linux systems the directory can be found at: /home/xxx/.genomecatpro/dev

Deletion of this file initiates the creation of a new configuration file

Why do CBS/HMM/RINGO do not work the first time after I have installed GenomeCAT?

These functions are implemented as external R packages, which are loaded into GenomeCAT from Bioconductor upon first usage of the R packages. For doing so GenomeCAT needs internet access.

What other reasons could cause problems with installation and updating of R packages?

If you do not have installed R in the course of the GenomeCAT installation, GenomeCAT may have not the rights to install or update R packages. GenomeCAT reports any failure to install these R packages in the module window. In this case you should install these packages by yourself.
If you rely on a preinstalled R environment it can also be that the R binaries are not accessible. In GenomeCATs installation directory, please edit the files setenv.sh or setenv.bat, respectively, to correct the path to R.
For example: R_BIN=c:\Program Files\R\bin

How can I improve the performance of GenomeCAT?

As of GenomeCAT version 4.4 the minimal and maximal RAM is determined and set automatically. If you want to increase memory (because your computer system is powerful enough to do so) you can change these settings in startGenomecat.bat or startGenomecat.sh ( you can find them in GenomeCATs installation directory).
On the example of WINDOWS startGenomecat.bat:

Original settings:

set /a min_memsize=%mem_size%/4
set /a max_memsize=%mem_size%/2

Replace "%mem_size%/4" bzw "%mem_size%/2" by the parameters you want and save the changes. For example:

set /a min_memsize=2g
set /a max_memsize=4g

In the example above you have changed memory size to 2Gb minimum and 4Gb maximum.
Please take care: if your system can not cope with your setting the start of GenomeCAT fails.
If Java cannot find enough memory it dispatches an exception message to the command line window as "java.lang.OutOfMemoryError: Java heap space ".
For more information see Oracle Doc: Tuning Tips for Heap Sizes.
